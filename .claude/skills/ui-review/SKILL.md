# UI 设计审查

像设计师一样审查前端页面的视觉质量。不是 CSS linter — 是评估"这个界面看起来是否精致、一致、专业"。

## 怎么审查

读取项目的设计系统（CSS 变量、组件类名），然后逐页审视模板，从以下四个维度评估。

### 1. 视觉层级

操作的重要性应该通过视觉重量体现。问自己：

- 页面上最醒目的元素是什么？它应该是最重要的操作吗？
- 同一行内的按钮，视觉重量是否与操作重要性成正比？（主要 CTA > 破坏性操作 > 次要操作 > 内联操作）
- 有没有"不该这么重"的按钮抢走了注意力？（如 toggle 开关比删除按钮更醒目）
- 状态指示和操作按钮是否混在一起造成视觉混乱？

**典型问题**: 填充按钮用于轻量 toggle、badge + button 堆叠在同一位置、所有按钮视觉重量相同。

### 2. 跨页面一致性

同类操作在不同页面应该看起来一样。问自己：

- "返回列表"在所有详情页是否使用相同的样式和符号？
- "删除"按钮在列表页和详情页是否一致？
- 分页控件在不同列表页是否统一？
- 复制/拷贝操作在所有页面是否使用相同的交互模式？

**典型问题**: 同一操作在 A 页面用 `btn-sm`、B 页面用 `btn`；返回按钮有的带 `←` 有的不带；分页有的用 `«»` 有的不用。

### 3. 精致度

细节决定质感。问自己：

- 按钮和相邻内容之间有没有适当的间距？（不应紧贴 `<code>`、`<input>` 或文字）
- 表格单元格内的元素排列是否整齐？（多个操作按钮之间是否有间距）
- 内联样式（`style="..."`）是否应该提取为 CSS 类？
- 符号使用是否统一？（箭头、连接符、省略号等不应混用不同写法）

**典型问题**: 按钮紧贴内容无间距、模板中大量 `style=""` 属性、同一页面混用 `→` 和 `←` 和 `»`。

### 4. 设计系统合规

项目定义了设计 token（CSS 变量），所有样式应从中派生。问自己：

- 颜色是否都使用了 `--color-*` 变量，而非硬编码 hex/rgb？
- 字号是否在类型比例尺上（`--text-*`），而非随意 rem 值？
- 间距是否在网格上（`--space-*`），而非 magic number？
- 按钮和图标是否使用了标准尺寸（`--btn-height*`、`--icon-*`）？

**典型问题**: CSS 中出现 `#333`、`font-size: 1.35rem`（不在比例尺上）、`padding: 0.6rem 2.4rem`（应使用变量）。

## 输出格式

按严重级别分组。每个发现用一句话描述问题，一行给出建议修复。

```
## UI 审查报告

### Critical
- **[页面/组件]** 问题描述 → 建议修复

### Warning
- **[页面/组件]** 问题描述 → 建议修复

### Nit
- **[页面/组件]** 问题描述 → 建议修复
```

## 审查原则

- 报告真正的视觉问题，不是代码风格偏好
- 如果差异有合理理由（如 primary 比 default 更重是有意为之），不要报告
- 优先报告用户能感知到的问题（视觉不一致 > 代码规范）
- 用设计师的语言描述问题（"这个按钮太抢眼了"），而非 linter 的语言（"line 42: hardcoded font-size"）
